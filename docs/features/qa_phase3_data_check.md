# 階段 3：資料驗證 (Data Verification)

## 📌 階段概述

**觸發時機**：Phase 2（資料填充）的所有必要條件達成後，全員同步進入 Phase 3

- 所有關鍵節點的 EFU 完整化已達成（向上、向側、向下三維度完整）
- 所有已派發的 Phase 2 任務均已回答完畢（或跳過）
- 系統檢測到「任務隊列為空」且「EFU 完成率 = 100%」

> ⚠️ **注意**：Phase 3 不由時間耗盡觸發。時間限制仍然存在（90/120/180 秒），但僅作為整場遊戲的最大時長上限。若 Phase 2 提前完成，Phase 3 可在時間限制內開始。

**玩家體驗**：無縫銜接，持續收到驗證性問題

- 玩家不會看到「階段切換」提示，問題風格從「填寫資料」轉為「驗證事實」
- 每道驗證題都有即時回饋（答對/答錯），讓玩家感受到遊戲的節奏變化
- 驗證過程中**不顯示**完整族譜結構

**系統目標**：

1. 透過主動出題驗證先前填寫的資料正確性
2. 發現並解決資料衝突（不同玩家對同一事實的認知差異）
3. 直接採用驗證後的正確答案即時更新族譜
4. 為後續揭曉階段（Phase 4）提供經過驗證的族譜資料

## 🔍 驗證出題系統

### 出題策略

驗證出題採用**抽樣驗證**策略，不追求 100% 覆蓋每個節點的每個屬性，而是在有限時間內儘量驗證最重要的資料：

| 策略面向     | 規則                                                              |
| ------------ | ----------------------------------------------------------------- |
| **覆蓋率**   | 目標 50-70% 的族譜節點與屬性                                      |
| **出題範圍** | 所有已填寫的節點（包括關鍵節點與旁系親屬）                        |
| **優先級**   | 關鍵路徑節點 > 多人填寫有分歧的屬性 > 僅單人填寫的屬性 > 旁系親屬 |
| **出題數量** | 依玩家人數與剩餘時間動態調整                                      |
| **持續出題** | Phase 3 不限定固定題數，在遊戲時間耗盡前持續出題                  |

#### 出題演算法（偽代碼）

```
function generateVerificationQuestions(familyTree, remainingTime):
  questionPool = []

  // Step 1：遍歷所有已填寫節點，產生候選題目
  for node in familyTree.filledNodes:
    questions = generateQuestionsForNode(node)
    questionPool.push(...questions)

  // Step 2：依優先級排序
  questionPool.sort(by: [
    node.isKeyNode DESC,          // 關鍵節點優先
    hasMultipleAnswers DESC,      // 多人分歧的屬性優先
    relationshipProximity ASC,    // 與回答者關係越近越優先
    randomShuffle                 // 同優先級內隨機
  ])

  // Step 3：依剩餘時間分配題目給各玩家
  for player in players:
    assignedQuestions = pickQuestions(questionPool, player, remainingTime)
    dispatch(player, assignedQuestions)

  // Step 4：持續出題直到時間耗盡
  while (gameTimer > 0):
    if (anyPlayerIdle):
      assignNextQuestion(idlePlayer)
```

### 派發規則

| 規則         | 說明                                                                         |
| ------------ | ---------------------------------------------------------------------------- |
| **迴避自填** | 優先將驗證題派發給**非填寫者**，避免「自己驗證自己」的無意義回答             |
| **親疏匹配** | 優先將題目派給與該節點**關係較近**的玩家（例如驗證「爺爺」的題目優先問孫子） |
| **均勻分配** | 避免所有題目集中在同一位玩家身上，維持參與感                                 |
| **不重複**   | 同一題目不重複派發給同一位玩家                                               |

## 🔄 多人驗證流程

### 核心機制

Phase 3 的核心驗證邏輯為**三方比對**：族譜現有資料、回答者 A 的答案、回答者 B 的答案。透過交叉驗證確定最終正確答案。

```
驗證流程圖：

  系統出題 → 派發給玩家 A
                    │
              ┌─────┴─────┐
              │            │
         A 答案 = 族譜   A 答案 ≠ 族譜
              │            │
          ✅ 驗證通過     ⚠️ 發現分歧
          A 得分          │
                     同題派發給玩家 B
                          │
                    ┌─────┴─────┐
                    │            │
               B 答案 = A 答案  B 答案 ≠ A 答案
                    │            │
              族譜有誤，更新     ┌────┴────┐
              A、B 皆得分       │          │
                           B = 族譜   三方各異
                              │          │
                         A 有誤      標記為衝突
                         B 得分      待人工審核
```

### 情況 A：答案與族譜一致（驗證通過）

玩家回答與族譜現有資料**相符**。

```
範例：
  族譜記錄：王大明的父親 = 王老先生
  系統問玩家 C：「王大明的父親是王老先生嗎？」
  玩家 C 回答：「是」

  → 驗證通過
  → 玩家 C 獲得驗證得分（+1 分）
  → 族譜資料不變，該屬性標記為「已驗證」
```

### 情況 B：答案與族譜不一致（轉發驗證）

玩家回答與族譜現有資料**不符**，系統無法判斷誰對誰錯，因此將同一題目轉發給另一位玩家進行第二次驗證。

```
範例：
  族譜記錄：王大明有 2 位小孩
  系統問玩家 A：「王大明有幾位小孩？」
  玩家 A 回答：「3 位」（與族譜不符）

  → 系統將同題轉發給玩家 B
  → 系統問玩家 B：「王大明有幾位小孩？」
```

#### 情況 B-1：第二位玩家與第一位玩家答案一致（族譜有誤）

**判定**：兩位玩家形成共識，族譜資料有誤。

```
  玩家 B 回答：「3 位」（與玩家 A 一致）

  → 結論：族譜的「2 位」有誤
  → 動作：即時更新族譜為「3 位」
  → 計分：玩家 A +1.5 分（發現錯誤獎勵）、玩家 B +1 分
```

#### 情況 B-2：第二位玩家與族譜一致（第一人答錯）

**判定**：第一位玩家回答有誤，族譜資料正確。

```
  玩家 B 回答：「2 位」（與族譜一致）

  → 結論：玩家 A 答案有誤，族譜正確
  → 動作：族譜不變，該屬性標記為「已驗證」
  → 計分：玩家 A 不得分（0 分）、玩家 B +1 分
```

#### 情況 B-3：三方答案各異（無法判定）

**判定**：三方答案互不相同，無法自動判斷正確性。

```
  族譜記錄：2 位
  玩家 A 回答：3 位
  玩家 B 回答：4 位

  → 結論：三方答案各異，無法自動解決
  → 動作：標記為衝突，維持族譜原值
  → 計分：所有相關玩家不得分也不扣分（0 分）
```

### 轉發派發規則

| 規則           | 說明                                                       |
| -------------- | ---------------------------------------------------------- |
| **排除原答者** | 轉發時不可再派發給第一位回答者                             |
| **排除原填者** | 優先排除 Phase 2 中填寫該屬性的玩家（因其答案已在族譜中）  |
| **親疏優先**   | 優先派發給與該節點關係較近的玩家                           |
| **快速需求**   | 轉發問題的優先級高於普通驗證題，插隊排入目標玩家的下一題   |
| **超時處理**   | 若轉發等待超過 15 秒無回應，標記為「未驗證」，維持族譜原值 |

## 📊 驗證計分機制

### 基礎得分

| 驗證結果                             | 得分    | 說明                           |
| ------------------------------------ | ------- | ------------------------------ |
| 答案與族譜一致（驗證通過）           | +1 分   | 確認族譜正確性                 |
| 答案與族譜不符，但經第二人確認為正確 | +1.5 分 | 發現並修正族譜錯誤，貢獻度更高 |
| 作為第二驗證者，確認第一人的答案     | +1 分   | 協助確認正確答案               |
| 作為第二驗證者，與族譜一致           | +1 分   | 維護族譜正確性                 |
| 答案錯誤（被第二人否決）             | 0 分    | 無懲罰，但不得分               |
| 三方各異（無法判定）                 | 0 分    | 無懲罰，標記為衝突             |

### 速度獎勵

| 回答時間   | 額外加分 |
| ---------- | -------- |
| 5 秒內回答 | +0.3 分  |
| 5-10 秒    | +0.1 分  |
| 超過 10 秒 | 無       |

### 一致性懲罰

若玩家在 Phase 3 的驗證答案與自己在 Phase 2 填寫的答案**矛盾**（自我否認），扣除：

```
一致性懲罰：-0.5 分

範例：
  Phase 2：玩家 A 填寫「爺爺的生日是 1950-01-01」
  Phase 3：玩家 A 驗證時回答「爺爺的生日不是 1950-01-01」

  → 玩家 A 收到一致性懲罰 -0.5 分
  → 但系統仍會將此視為合法回答，進行後續轉發驗證
```

## ⚡ 族譜即時更新機制

### 更新規則

Phase 3 驗證過程中，族譜會被**即時更新**（而非等到 Phase 4 才更新）：

| 觸發條件                | 更新動作                         | 通知                   |
| ----------------------- | -------------------------------- | ---------------------- |
| 驗證通過（答案 = 族譜） | 無更新，僅標記該屬性為「已驗證」 | 無                     |
| 兩人一致，族譜有誤      | 更新族譜為兩人的共識答案         | 通知旁觀者更新族譜顯示 |
| 三方各異                | 維持原值，標記為「衝突」         | 通知旁觀者顯示衝突標記 |

### 更新記錄格式

每次族譜更新都會記錄：

```json
{
  "update_id": "verify_update_001",
  "node_id": "virtual_grandfather_A",
  "attribute": "childrenCount",
  "old_value": 2,
```

## ⚠️ 衝突記錄與標記

### 衝突定義

在 Phase 3 中，衝突專指**三方比對無法得出共識**的情況：

| 衝突類型     | 觸發條件                                       | 嚴重性 |
| ------------ | ---------------------------------------------- | ------ |
| 三方各異衝突 | 族譜值、玩家 A、玩家 B 三方答案互不相同        | 高     |
| 自我否認衝突 | 玩家在 Phase 3 否認自己在 Phase 2 填寫的資料   | 中     |
| 邏輯矛盾衝突 | 驗證結果導致族譜出現邏輯矛盾（如循環親子關係） | 高     |

### 衝突記錄格式

```json
{
  "conflict_id": "conflict_phase3_001",
  "node_id": "virtual_grandfather_A",
  "attribute": "name",
  "conflict_type": "three_way_disagreement",
  "severity": "high",
  "tree_value": "王大明",
  "player_answers": [
    { "playerId": "p1", "answer": "王大民", "timestamp": "..." },
    { "playerId": "p3", "answer": "王大銘", "timestamp": "..." }
  ],
  "resolution": "pending",
  "created_at": "2026-02-27T10:35:00Z"
}
```

### 未解決衝突的處理

Phase 3 結束時，未解決的衝突將帶入 Phase 4（揭曉階段）：

- 衝突節點在族譜中以**紅色邊框**標記
- 玩家可在揭曉頁面點擊查看衝突詳情
- 系統暫時維持**族譜原值**作為顯示用

## ✅ 完整性驗證

Phase 3 結束前（遊戲時間耗盡時），系統執行最終完整性驗證：

### 檢查清單

✓ **檢查 1**：所有玩家都有父親與母親節點 → 確保基本 EFU 結構完整

✓ **檢查 2**：所有父母節點都有姓名與性別 → 確保基本資料不缺失

✓ **檢查 3**：所有在 Phase 1 確認有關係的玩家在同一棵族譜樹上 → 確保連通性

✓ **檢查 4**：Phase 3 產生的族譜更新未造成邏輯矛盾 → 確保一致性

✓ **檢查 5**：衝突數量統計 → 為揭曉階段提供警告資訊

## 🎯 特殊情況處理

### 情況 1：剩餘時間不足

**現象**：Phase 2 完成時，遊戲剩餘時間極短（例如 < 10 秒）

**處理方式**：

- Phase 3 仍會啟動，但可能只派發 1-2 道驗證題
- 短時間內優先派發**關鍵路徑節點**的驗證題
- 來不及完成的驗證題不計分、不扣分

### 情況 2：所有玩家快速完成

**現象**：Phase 2 很早完成，Phase 3 有充裕時間

**處理方式**：

- 系統持續出題，可對同一節點進行多次交叉驗證
- 已驗證過的屬性可能再次被抽到（以不同問法）
- 覆蓋率可能超過 70%

### 情況 3：轉發找不到可用玩家

**現象**：需要轉發驗證，但所有其他玩家都已斷線或忙碌

**處理方式**：

- 等待最多 15 秒，若仍無可用玩家
- 標記為「未驗證」，維持族譜原值
- 不對任何玩家計分或扣分

### 情況 4：遊戲只有 2 位玩家

**現象**：轉發驗證時只能轉給唯一的另一位玩家

**處理方式**：

- 正常運作，轉發給唯一的另一位玩家
- 若兩人答案與族譜三方各異，直接標記衝突
- 2 人遊戲的衝突解決更依賴「直系關係人優先」原則

### 情況 5：Phase 3 期間玩家斷線

**現象**：玩家在驗證過程中斷線

**處理方式**：

- 已回答的驗證題正常計分
- 未回答的驗證題取消，不計分不扣分
- 正在等待轉發的題目改派其他玩家
- 若斷線玩家重連，繼續收到新的驗證題

---

## 📋 Phase 3 結束條件

Phase 3 於以下任一條件成立時結束，進入 Phase 4（揭曉階段）：

1. **遊戲時間耗盡**（90/120/180 秒總時長到達）
2. **所有已填寫節點的所有屬性都已驗證**（極少見，但理論上可能）

結束時，系統封存所有驗證紀錄與衝突標記，將最終版族譜交付 Phase 4 進行揭曉。
