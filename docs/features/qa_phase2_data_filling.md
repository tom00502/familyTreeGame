# 階段 2：資料填充 (Data Completion)

## 📌 階段概述

**核心目標**：透過資料填充，完成所有**關鍵節點**的 **EFU 完整化**（Essential Family Unit），進而完成最小可行族譜 (MVFT)

**EFU 完整化定義**：每位關鍵節點都擁有完整的三維度親屬資訊：

- ✅ **向上維度**：父親、母親
- ✅ **向側維度**：配偶（若已婚）
- ✅ **向下維度**：所有子女

詳見 [concepts/mvft.md 中的 EFU 定義](../concepts/mvft.md#核心家庭單元-essential-family-unit-efu)。

**玩家體驗**：持續回答「虛擬節點資訊」的各種題目（姓名、性別、生日、親屬確認等），但無法看到完整族譜結構。玩家回答的每個問題都獨立且互不相關，不知道這些答案如何拼湊成族譜

**系統目標**：利用玩家的分散答案，透過「獲取或建立」(Get or Create) 邏輯逐步構建完整的虛擬節點家族樹，並完成所有關鍵節點的 EFU 完整化

## � 核心術語定義

### 骨架族譜 (Skeleton MVFT)

**定義**：族譜圖在「階段一：關係確認」結束後產出的中間態邏輯模型。具備完整的圖論結構與邏輯一致性，但節點資料主要由虛擬節點構成。

**特徵**：

| 面向                              | 狀態      |
| --------------------------------- | --------- |
| 結構完整性 (Structural Integrity) | ✅ 完整   |
| 資料完整性 (Data Integrity)       | ❌ 不完整 |

**組成元素**：

- **玩家節點** (`isVirtual: false, isPlayer: true`)：房間內真實連線的玩家
- **虛擬關鍵節點** (`isVirtual: true, isKeyNode: true`)：位於關鍵路徑上，連結玩家的祖先或中間人
- **預製邊 (Prefab Edges)**：基於稱謐解析出的邏輯關係線，待名稱與順序填寫

**作用**：作為「關係連連看」（階段一）與「資料搶答」（階段二）之間的橋樑，定義「哪些資訊是必須填寫的」。

### 關鍵節點 (Key Node) vs 葉子人物 (Leaf Member)

**關鍵節點**：位於玩家間關鍵路徑上的節點。

- **EFU 完整化要求**：⚠️ **必須完整**（父、母、配偶、子女都必須有記錄）
- **任務派發優先級**：最高（Level 1-3）
- **判定方式**：使用 BFS/DFS 演算法尋找玩家間最短路徑上的所有節點
- **例子**：玩家的父親、祖父母、配偶、子女

**葉子人物**：EFU 中的相關人物（父、母、配偶、子女），但本身不在任何玩家間關鍵路徑上的人。

- **EFU 完整化要求**：❌ **不需要**（只需姓名、性別、基本屬性）
- **任務派發優先級**：次之（Level 4-6）或可能不派發
- **擴張方式**：**終端擴張**（Terminal Expansion），不對其親屬進行延伸追蹤
- **例子**：玩家父親的父親（玩家祖父母），玩家配偶的親友，非關鍵路徑上的堂兄弟

## �📝 資料填充任務系統

### 概述

資料填充階段透過五類任務類型的組合，逐步實體化虛擬節點並完成 MVFT。各任務透過**優先級分級系統**與**獨佔提問機制**進行派發，確保效率與資料準確性。
**任務目標**：所有任務都圍繞「完成關鍵節點的 EFU 三維度」展開。

### 📐 EFU 三維度任務分解

資料填充任務圍繞 EFU 的三個維度組織，每個維度對應不同的優先級與任務類型：

| EFU 維度         | 優先級 | 核心任務 | 任務類型                     | 說明                                   |
| ---------------- | ------ | -------- | ---------------------------- | -------------------------------------- |
| **向上（父母）** | L1     | 路徑閉合 | 向上追溯 + 節點命名          | 確認關鍵節點的父親、母親，建立代際連結 |
| **向側（配偶）** | L2     | 橫向補齊 | 屬性填充（性別、姓名）       | 確認關鍵節點的配偶，完成家庭核心單元   |
| **向下（子女）** | L2     | 下代確認 | 節點命名 + 屬性填充          | 確認關鍵節點的所有子女，完成世代延續   |
| **旁系補充**     | L4-6   | 可選擴張 | 節點匯聚、排序確認、屬性補充 | 補充非關鍵路徑上的親屬資訊（可選）     |

## 📝 五類任務類型

### 任務類型 1：節點命名 (Node Naming)

**[EFU 維度] 向上 / 向下 | [優先級] L1 / L2**

**觸發條件**：虛擬節點存在但姓名為空

**提問模板**：

```
「你的 {node_label} 叫什麼名字？」
「請問{node_name}的父親叫什麼名字？」
「有人記得這位{node_label}的名字嗎？」
```

**玩家操作**：

- 輸入 2-20 個中文字元的姓名
- 可選擇「不知道」（任務重新分派）
- 可選擇「跳過」（任務延後）

---

### 任務類型 2：屬性填充 (Attribute Filling)

**[EFU 維度] 向上 / 向側 / 向下 | [優先級] L2 / L3**

**觸發條件**：節點已有姓名，但缺少性別、生日或年齡資訊

**提問模板**：

```
「{node_name}是男性還是女性？」
「有人記得{node_name}的生日或大約幾歲嗎？」
「{node_name}大約是幾年出生？」
```

**玩家操作**：

- 性別：二選一（男性/女性）或「不知道」
- 生日：輸入 YYYY/MM/DD 或「不知道」
- 年齡：輸入數字或「不知道」

---

### 任務類型 3：向上追溯 (Upward Tracing)

**[EFU 維度] 向上 | [優先級] L1**

**觸發條件**：當前節點是關鍵路徑上的節點，但其父親或母親為空

**提問模板**：

```
「{node_name}的父親是誰？」
「{node_name}的母親叫什麼名字？」
「有人知道{node_name}大概的上輩是誰嗎？」
```

**玩家操作**：

- 若知情：輸入虛擬節點該祖代親戚的名字，或確認已有的名字
- 若不知道：選擇「不知道」
- 若涉及多個可能的答案：選擇「不確定」（標記為待確認）

---

### 任務類型 4：節點匯聚 (Node Convergence)

**[EFU 維度] 向上 / 向側 / 向下（衝突驗證） | [優先級] L2**

**觸發條件**：系統偵測到兩條不同路徑可能指向同一個人

**提問模板**：

```
「玩家 B，你的{node_label}是『{known_name}』嗎？」
「{node_A}和{node_B}是同一個人嗎？」
```

**背景**：

- 不同玩家可能從不同途徑提到同一個人的名字
- 或系統根據親屬關係推導出兩條路徑應該匯聚
- 此任務確認這兩條路徑是否指向同一人

**玩家操作**：

- 是／否 選擇
- 或「不確定」（標記為待確認）

---

### 任務類型 5：排序確認 (Age Ordering)

**[EFU 維度] 向上 / 向側 / 向下（橫向比較） | [優先級] L3-L5**

**觸發條件**：系統需要確認多個同輩或相鄰代數的節點之間的年齡大小關係

**提問模板**：

```
「{node_A}與{node_B}誰的年紀比較大？」
「按出生順序，下列人誰最年長？{list}」
```

**玩家操作**：

- 選擇更年長者
- 若無法判斷：選擇「不知道」

## 🎯 任務派發策略

### 優先級分級系統

系統使用**多維度評分模型**決定將哪個任務分派給哪位玩家。此表整合了任務優先級與 EFU 完整化目標的對應關係：

| 優先級 | EFU 完整化階段 | 任務類型             | EFU 維度       | 目標                   | 權重 | 例子                         |
| ------ | -------------- | -------------------- | -------------- | ---------------------- | ---- | ---------------------------- |
| **L1** | ✅ 必需        | 向上追溯 + 節點命名  | 向上           | 路徑閉合（父母確認）   | 100% | 填寫玩家父母、祖父母資訊     |
| **L2** | ✅ 必需        | 節點匯聚 + 屬性填充  | 向側/向下      | 橫向補齊（配偶、子女） | 80%  | 確認玩家的配偶、確認「爺爺」 |
| **L3** | ✅ 必需        | 屬性填充（關鍵路徑） | 向上/向側/向下 | 補齊關鍵路徑詳細資訊   | 60%  | 填寫父親的生日、配偶資訊     |
| **L4** | ❌ 可選        | 節點命名（旁系）     | —              | 擴張旁系親屬資訊       | 40%  | 填寫伯父、堂兄弟的名字       |
| **L5** | ❌ 可選        | 排序確認             | —              | 邏輯驗證與視覺排序     | 30%  | 確認叔叔與父親誰更年長       |
| **L6** | ❌ 可選        | 屬性填充（旁系）     | —              | 補充選擇資訊           | 20%  | 填寫堂弟的生日               |

**說明**：

- **L1-L3（藍色必需）**：與關鍵節點的 EFU 完整化直接相關，確保第二階段最核心的目標達成
- **L4-L6（灰色可選）**：補充非關鍵路徑的親屬資訊，提升族譜豐富度但非必需

### 優先級計算公式

系統以**多維度評分模型**為每個「玩家 × 任務」配對計算分數，選出最高分的配對進行派發。優先級分數與 EFU 完整化的優先級緊密掛鉤：

**L1-L3 任務（EFU 完整化必需）** 的優先級分數計算：

```
Priority Score = (0.6 × Proximity) + (0.2 × ContributionScore) + (0.1 × RandomValue) - (0.1 × SkipCount)
```

**L4-L6 任務（可選擴張）** 的優先級分數會被進一步衰減，確保 L1-L3 任務始終優先派發。

| 維度                        | 權重 | 說明                                                                                |
| --------------------------- | ---- | ----------------------------------------------------------------------------------- |
| Proximity（關係鄰近度）     | 0.6  | BFS 距離越近分數越高，見「規則 1：關係親近優先」。與 EFU 維度的關聯性越強，分數越高 |
| ContributionScore（貢獻度） | 0.2  | 已完成任務數比例，確保參與機會均衡，防止單人壟斷題目                                |
| RandomValue（隨機因子）     | 0.1  | `[0, 1]` 隨機數，增加多樣性，避免演算法過度決定論                                   |
| SkipCount（跳過次數懲罰）   | 0.1  | 此任務已被跳過的次數，每跳過一次降低分數。L1-L3 任務被跳過時懲罰更重                |

> 💰 **分數如何計算** → 請查閱 [scoring_system.md](./scoring_system.md)

### 派發算法規則

#### 規則 1：關係親近優先

任務優先派發給與該節點**關係最近**的玩家：

```
例：需要填寫「玩家 A 的祖父」的姓名

派發優先級：
  1️⃣ 玩家 A 本身（親近度最高）
  2️⃣ 玩家 A 的父親（如果在遊戲中）
  3️⃣ 玩家 A 的兄弟姊妹（玩家）
  4️⃣ 玩家 A 的堂兄弟（玩家）
  5️⃣ 任何其他玩家（親近度最低）
```

#### 規則 2：獨佔提問 (Exclusive Quizzing)

**核心機制**：同一個任務在同一時刻**只派發給一位玩家**

**詳細邏輯**：

1. **單一派發**
   - 系統將任務 T 分派給玩家 P（根據上述優先級算法選出）
   - 只有玩家 P 在其螢幕上看到任務 T
   - 該任務從其他玩家的潛在題目池中移除

2. **鎖定機制**
   - 玩家 P 開始輸入時，系統發送 `task_lock` 事件
   - 其他玩家收到通知：「{玩家名} 正在填寫該資訊」
   - 任務從其他玩家的潛在題目池中移除
   - 其他玩家無法提交關於此任務的答案

3. **任務流轉**
   - **完成**：玩家 P 提交答案 → 任務完成，累積分數
   - **跳過**：玩家 P 選擇「跳過」 → 任務優先級降低，重新分派給其他玩家
   - **超時未動作**：超過時間限制 → 自動作為「跳過」處理

4. **狀態轉換圖**

```
未分派
  ↓
[分派給玩家A]
  ↓
已分派 ← [玩家A看到任務]
  ↓
  ├─→ [玩家A開始輸入] → 鎖定中 ↘
  │                          ├─→ [提交] → 完成（移出）
  │                          └─→ [跳過] → 優先級↓，重新分派
  │
  └─→ [超時] → 優先級↓，重新分派
```

#### 規則 3：跳過流轉與優先級衰減

玩家跳過或超時後，系統將任務**重新進入任務池**：

```
初始派發：玩家A → [跳過] → 玩家B
玩家B完成或跳過：玩家B → [跳過] → 玩家C
依次流轉，直到完成或時間耗盡
```

**優先級衰減**：

- 每被跳過一次，優先級分數 × 0.8（衰減 20%）
- 被跳過 5 次或以上的任務，自動降至最低優先級

### 核心概念：「獲取或建立」邏輯

此邏輯是系統的「生長引擎」，將抽象稱謝轉化為圖形結構：

```
當玩家說「A 是我的堂兄」時：
  系統解析路徑：A → P(男) → P(男) → C(男) → 我
  沿路徑走訪：
    1. 檢查是否有「我的父親」節點 → 若無則建立（虛擬）
    2. 檢查是否有「我的祖父」節點 → 若無則建立（虛擬）
    3. 檢查是否有「祖父的兒子（我的伯叔）」節點 → 若無則建立（虛擬）
    4. 確認此伯叔是否有兒子 → 若無則建立
  最終：A 與「伯叔的兒子」匹配 → 關係確認
```

### 原子動作定義

系統的三個基本方向動作（Atomic Operations）：

| 動作 | 符號 | 含義           | 例子                  |
| ---- | ---- | -------------- | --------------------- |
| 向上 | P    | Parent（往上） | P = 父親，P² = 祖父   |
| 向下 | C    | Child（往下）  | C = 兒子，C² = 孫子   |
| 水平 | S    | Spouse（配偶） | S = 配偶（丈夫/妻子） |

### 路徑解析範例表

| 稱謝   | 方向 | 路徑                    | 說明                     |
| ------ | ---- | ----------------------- | ------------------------ |
| 爸爸   | 父系 | P                       | 直接向上一代             |
| 爺爺   | 父系 | P²                      | 向上兩代，父親的父親     |
| 曾祖父 | 父系 | P³                      | 向上三代                 |
| 伯父   | 父系 | P(兄) = P.Sibling(男)   | 父親的男性兄長           |
| 叔叔   | 父系 | P(弟) = P.Sibling(男,Y) | 父親的男性幼弟           |
| 堂兄   | 父系 | P.P.C(男,O) = P².C(男)  | 祖父的子女的男性長子     |
| 表兄   | 母系 | M.P.C(男,O) = M.P².C    | 母親之父的子女的男性長子 |
| 兒子   | 子女 | C                       | 直接向下一代             |
| 孫子   | 子女 | C²                      | 向下兩代                 |
| 配偶   | 配偶 | S                       | 水平方向，配偶本人       |

| 女婿   | 配偶 | C(女).S                 | 女兒的配偶               |

**註**：路徑符號的通用規則

- `P` = 父親或母親（由方向確定）
- `M` = 母親（特別標記時）
- `C(gender, order)` = 子女，可指定性別與出生順序
- `.` = 路徑連接符
- `Sibling(gender)` = 兄弟姊妹，可過濾性別

## 🔄 系統實作邏輯流程

### 流程 1：路徑實體化 (Instantiation)

系統掃描第一階段結果，根據「稱謝路徑表」自動建立虛擬節點：

```
輸入：第一階段結果
  - 玩家 A: 玩家 B 是我的堂兄
  - 玩家 C: 玩家 B 是我的表兄

過程：
  1. 解析「堂兄」路徑：P².C → 需要 A 的祖父級節點
  2. 解析「表兄」路徑：M.P².C → 需要 C 的祖父級節點（母親那邊）
  3. 檢查虛擬節點集合，不存在則建立：
     - 虛擬節點「A 的祖父」
     - 虛擬節點「A 的祖父的兒子（伯叔）」
     - 虛擬節點「C 的外公」（母親的父親）
     - 虛擬節點「C 的外公的兒子（舅舅）」
  4. 標記「A 是虛擬節點『伯叔』的兒子」
  5. 標記「B 是虛擬節點『伯叔』的兒子」

輸出：
  - 新虛擬節點列表
  - 路徑實體化完成，準備進入任務分派
```

### 流程 2：任務分派 (Dispatching)

根據優先級算法，將「命名」與「追溯」任務丟入 Task Queue：

```
入隊任務（優先級排序）：
  1. 命名「A 的祖父」(優先級: 100, 派發給：A)
  2. 命名「A 的祖母」(優先級: 100, 派發給：A)
  3. 追溯「祖父的兒子」(優先級: 90, 派發給：A 或 C)
  4. 命名「外公」(優先級: 100, 派發給：C)
  ...
```

### 流程 3：節點去重與匯聚 (Deduplication)

當不同玩家提到同一個名字或透過匯聚問題確認身分後，系統合併節點 ID：

```
示例：
  - A 說「我的堂兄是王二明」
  - B 說「B 的舅舅是王二明」
  → 系統自動匯聚：「王大明」= 「王二明」（可能同一人）
  → 詢問相關玩家確認
  → 一旦確認，合併兩個虛擬節點 ID
```

**去重規則**：

- 同名同姓 → 透過年齡或配偶資訊進行二次確認
- 名字變種（「大明」 vs「王大明」）→ 記錄為疑似同一人，待確認
- 衝突資訊（同一人不同年齡）→ 標記為「衝突」，移交第三階段處理

### 流程 4：動態更新

一旦節點有了新資訊，所有相關問題自動更新：

```
初始問題（節點暫無姓名）：
  「你的父親叫什麼名字？」

玩家填寫後：
  「你的父親叫『王大明』，他的生日是？」

再填寫性別後：
  「王大明（男性）是你的父親，請確認他的生日...」

實時影響其他問題：
  節點匯聚問題自動更新：
    「玩家 B，你的爺爺是『王金剛』嗎？」（之前展示的是虛擬節點 ID）
```

## ⚠️ 邊界情況處理

### 情況 1：多重婚姻與複雜關係

**現狀**：MVP 暫不處理再婚或領養，以一夫一妻制的直系血親為主

**處理方式**：

- 系統假設每位玩家有唯一的父親與母親（虛擬節點）
- 若玩家間關係涉及再婚（如「我是同父異母的兄弟」），暫時記錄為「待確認」狀態
- 在階段三進行人工審核或特殊標記

**EFU 完整化的影響**：

- 多重婚姻導致 EFU 向側（配偶）維度不完整時，標記為「不適用」而非「不完整」
- 系統仍應確保向上、向下維度的完整性

### 情況 2：同名同姓誤判

**現象**：家族中有兩位「王小明」，系統可能誤判

**處理方式**：

- 系統自動進行二次確認：詢問「王小明」的年齡、配偶名字等額外信息
- 若仍無法區分，標記為「疑似同一人」，移交第三階段人工處理
- 結果儲存為：`node_merge_candidates: [node_id_1, node_id_2]`

**EFU 完整化的影響**：

- 同名誤判可能導致虛擬節點重複，影響 EFU 的完整度判定
- 衝突解決後，系統應重新驗證相關節點的 EFU 完整性

### 情況 3：衝突標記

**現象**：玩家 A 與 B 對同一虛擬節點的資訊（如生日）回答不一致

**處理方式**：

- 系統**不應卡住**，繼續進行遊戲
- 將衝突標記為：`conflict: { node_id: xxx, field: 'birthday', values: ['1980-01-01', '1980-01-02'], sources: [playerA, playerB] }`
- 轉交至第三階段驗證與人工處理

**EFU 完整化的影響**：

- 即使資訊衝突，只要節點本身存在且有姓名、性別，仍視為「部分完整」
- 完整衝突解決後，重新評估節點的 EFU 完整度

### 情況 4：EFU 不完整的特殊場景

**現象**：某些關鍵節點因玩家資訊不足，導致 EFU 某維度無法完整

**常見情況**：

- **向上維度缺失**：玩家不知道親生父親或母親
- **向側維度缺失**：玩家未婚或配偶不是直系親屬
- **向下維度缺失**：玩家無子女

**處理方式**：

| EFU 維度         | 完整性要求                   | 處理邏輯                                                                                         |
| ---------------- | ---------------------------- | ------------------------------------------------------------------------------------------------ |
| **向上（父母）** | ⚠️ 必需                      | 若玩家確實不知道親生父母，標記為「孤立節點」，不能向上追溯；若系統推導出但玩家遺漏，派發補充任務 |
| **向側（配偶）** | ✅ 有則完整，無則視為已婚=否 | 未婚玩家的向側維度視為「完整」（無配偶）                                                         |
| **向下（子女）** | ✅ 有則完整，無則視為無子女  | 確認玩家的所有子女；若確認「無子女」，向下維度視為「完整」                                       |

**EFU 完整化判定更新**：

```
EFU 完整 = (有父親或「孤立」) AND (有母親或「孤立」)
         AND (有配偶或未婚)
         AND (有所有子女或已確認「無子女」)
```

### 情況 5：玩家離線與重連

| 情況     | 處理方式                                 |
| -------- | ---------------------------------------- |
| 玩家離線 | 標記為離線，暫停派題給他，已派題重新分派 |
| 玩家重連 | 恢復線上狀態，未完成的任務仍可繼續作答   |

### 情況 6：任務池耗盡

```
IF TaskQueue 為空 AND 所有玩家都完成分派:
  └─ 進入「大揭曉」階段
  └─ 不再派題，等待遊戲結束
```

### 情況 6：超時自動處理

```
IF 玩家 P 獲派任務 T，30 秒無反應:
  ├─ 自動作為「跳過」處理
  ├─ 任務 T 重新進入佇列
  └─ 優先級 Priority(P) 降低 20%
```

**EFU 完整化的影響**：

- 超時跳過的 L1-L3 任務，會影響 EFU 完整度達成進度
- 系統應在時間結束後，根據未完成的 L1-L3 任務評估 EFU 的「實際完整度」

### 情況 7：關係鎖定

```
IF 玩家 A & B 的關係已確認:
  └─ 系統永久記錄該關係
  └─ 不再提問同一對玩家的關係
  └─ 若後續有矛盾提交，觸發衝突解決
```

### 情況 9：驗證階段中的「待確認」狀態

```
IF 玩家在驗證階段選擇「不確定」:
  ├─ 將該節點標記為「待確認」
  ├─ 稍後可由其他玩家驗證或確認
  └─ 遊戲結束時仍未確認的節點會在「大揭曉」中標註
```

**EFU 完整化的影響**：

- 「待確認」狀態的節點，仍視為「部分完整」（有節點存在，但資訊待驗證）
- 不影響 EFU 完整度的計算，但會標記為「低信心」

## ⚖️ 衝突檢測與解決

### 衝突場景

**場景 1：對同一個人的描述矛盾**

```
節點 V1 (祖父)
├─ 玩家 A 填寫：姓名 = "王小明"，性別 = "男"，生日 = "1945/01/01"
├─ 玩家 B 稍後填寫：姓名 = "王曉民"，性別 = "女"，生日 = "1945/06/06"
→ 檢測衝突：姓名、性別、生日都不匹配
```

**場景 2：關係不一致**

```
A 說：C 是我的兒子
B 說：C 是我的兄弟
→ DAG 驗證時發現環路或矛盾
```

### 衝突解決流程

1. 衝突的雙方（或多方）同時收到衝突任務
2. 各自從選項中選擇一個答案（若選擇一致 → 衝突解決）
3. 若選擇仍有分歧 → 觸發「人工確認」（由房主或在場長輩決定）

**衝突任務範例**：

```
「關於這位祖父，有兩個不同答案，請選擇你認為正確的：」
  ├─ 同意 A 的答案（性別 = 男）
  ├─ 同意 B 的答案（性別 = 女）
  └─ 都不確定，標記為「待確認」
```

### 衝突檢測觸發點

1. **新資料提交時**（資料補齊階段）
   - 檢查新資料是否與已有資料衝突
   - 如果衝突，立即觸發「衝突解決任務」

2. **驗證階段**（資料驗證階段）
   - 玩家驗證已完成的資料時發現不一致
   - 觸發「衝突解決任務」

3. **DAG 驗證時**
   - 檢查新關係是否造成環路或矛盾
   - 如果發現，停止確認，改為衝突解決

### 衝突對 EFU 完整化的影響

**EFU 維度完整性在衝突存在時的處理**：

| 衝突類型          | EFU 維度影響     | 處理邏輯                                                                    |
| ----------------- | ---------------- | --------------------------------------------------------------------------- |
| **姓名/性別衝突** | 不影響完整性判定 | 保留兩個答案版本，標記為「待驗證」，不視為 EFU 不完整                       |
| **親屬關係衝突**  | 影響路徑完整性   | 若同一節點有多個可能的父親/母親，則向上維度視為「待確認」，暫時不計入完整度 |
| **邏輯矛盾衝突**  | 影響全局 EFU     | 環路或嚴重矛盾的情況，标记整個子树為「待驗證」，不計入 EFU 完整度           |

**衝突解決後的 EFU 重新評估**：

- 衝突確認後，系統應重新驗證受影響節點的 EFU 完整度
- 若衝突解決導致新的虛擬節點產生，繼續派發相關任務補齊 EFU

## 💾 階段二完成條件

**系統定義**（由後端自動判定）：

### ✅ EFU 完整化達成條件

此階段的核心目標是達成所有關鍵節點的 **EFU 完整化**。系統應驗證以下條件：

1. ✓ **向上維度**：所有關鍵節點都有父親與母親虛擬節點（或明確標記為「孤立」）
2. ✓ **向側維度**：所有已婚的關鍵節點都有配偶記錄；已確認單身的節點視為完整
3. ✓ **向下維度**：所有關鍵節點都確認了其所有子女（包括「無子女」的狀況）
4. ✓ **資料完整性**：所有EFU中的成員都至少有姓名與性別
5. ✓ **路徑完整性**：所有在第一階段確認有關係的玩家都在同一棵族譜樹上（連通性滿足）
6. ✓ **衝突解決**：沒有未解決的關鍵衝突（例如同一玩家指向兩個不同的祖父，或性別矛盾）

### 📊 EFU 完整化進度指標

後端應追蹤以下指標：

- **完整度百分比** = (已完成EFU的關鍵節點數 / 總關鍵節點數) × 100%
- **維度完整度** = 向上完成度 × 向側完成度 × 向下完成度
- **時間進度** = 已耗時秒數 / 總秒數

當「完整度百分比」≥ 100% 時，EFU 完整化達成。系統仍應繼續派發 L4-L6 任務直到時間耗盡。

**玩家感知**（由前端判定）：

❓ 玩家**無法感知**階段二何時完成

- 系統繼續派發新題目，即使 EFU 完整化已達成
- 題目內容從「必需」(L1-L3) 逐漸轉向「補充」(L4-L6)
- 當時間耗盡，自動進入階段三

## 📋 玩家操作指南

### 在階段二中，玩家可以：

✅ **快速回答**：盡可能多地填寫虛擬節點資訊

✅ **部分填寫**：每次只填一個欄位（姓名、性別、生日）

✅ **跳過問題**：選擇「跳過」將題目重新排隊

✅ **標記不知道**：選擇「不知道」表示無法確認

✅ **修正答案**：最後確認前可修改（全部提交後無法修改）

❌ **不能看族譜**：無法查看族譜的完整結構（保持神秘感）

❌ **不能搶答**：無法回答他人獨佔的任務（鎖定機制保護）

❌ **不能修改已提交的答案**：提交後無法回溯
